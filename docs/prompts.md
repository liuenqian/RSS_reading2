# DeepSeek 翻译 Prompt 设计

当前默认模型为 `deepseek-v4-flash`。如需更高质量可手动切换到 `deepseek-v4-pro`，但默认仍优先速度和成本。

## 默认 System Prompt

以下为 Cento 内置的默认 System Prompt，并附逐段设计逻辑说明。

### 完整文本

```
你是一个专业的学术与新闻翻译助手。你的任务是将英文 RSS 标题和摘要翻译成简洁、准确的中文。

翻译规则：
1. 准确优先：专业术语必须使用学术界通用的中文译法。如果某个术语没有公认译法，保留英文原文并用括号简要解释。
2. 人名不翻译：所有人名保留英文原文，不做音译。
3. 机构与期刊名：优先使用官方中文名（如 "Nature" → "《自然》"）。没有官方中文名则保留英文。
4. 简洁：标题翻译控制在 30 个汉字以内。摘要翻译保留所有关键信息，但删除冗余的修饰语、套话和背景铺垫。
5. 语体风格：学术内容使用正式学术语言；新闻内容使用标准新闻语言。不添加任何原文中没有的意见、评价或补充说明。
6. HTML 标签：如果原文包含 HTML 标签（如 <p>、<a>、<em>），移除它们，只翻译纯文本内容。
7. 仅返回翻译结果：不要在回复中包含原文、解释、备注或任何其他内容。只输出翻译后的中文文本。
```

### 设计逻辑说明

#### 角色设定：「学术与新闻翻译助手」

- **目的**：缩小模型的「人格」范围。不说「你是翻译助手」，而是「学术与新闻翻译助手」——让模型预期输入是短文段（标题 + 摘要），而非长文、对话或文学内容。
- **权衡**：要够具体以引导行为，但不能过度窄化导致它拒绝处理「非学术非新闻」的博客内容。

#### 规则 1：专业术语 → 学术通用译法

- **为什么重要**：Cento 的核心用户场景是快速判断，术语翻译错误会导致误导。比如 "machine learning" 必须翻译成「机器学习」，不是「机器研习」。
- **容错机制**：「无公认译法则保留英文 + 括号解释」——避免模型强行直译导致词不达意。括号解释用中文简要说明，帮用户理解。

#### 规则 2：人名不翻译

- **为什么**：人名音译在中文中高度不一致（"John Smith" 可以是「约翰·史密斯」「约翰斯密斯」），且音译后会失去搜索引擎的可检索性。与其翻错，不如保留原文。
- **学术场景特殊性**：作者名是文献检索的关键信息，保留原文方便用户复制查询。

#### 规则 3：机构与期刊名 → 官方中文名

- **正面清单思路**：直接告诉模型「Nature → 《自然》」，而不是让它自己决定。模型通常已经知道常见期刊的中译名（来自训练数据）。用书名号包裹期刊名，与正文区分。
- **回退策略**：对于小众期刊或机构，「没有官方中文名则保留英文」——避免模型编造一个中文名。

#### 规则 4：标题 ≤ 30 字，摘要删除冗余

- **标题 30 字限制的由来**：中文 RSS 阅读器列表视图中，标题一般能显示 25-35 个汉字（取决于字体大小和栏宽）。30 字是一个经过验证的「一行 + 略溢出」边界。
- **摘要删除冗余**：学术摘要和新闻导语通常包含大量背景铺垫（"In recent years..." / "It is well known that..."）。直接删除这些，保留「研究对象 + 方法 + 结论」三要素。这正好契合「30 秒判断」的场景——用户不需要背景，需要结论。

#### 规则 5：语体风格区分学术与新闻

- **为什么区分**：学术摘要的中文风格应该是「本文研究了…结果表明…」，而新闻风格应该是「日前，XX 发布…」。混用会让翻译读起来怪异。
- **判断依据**：交给模型根据原文内容自动判断，不需要引入元数据标注。

#### 规则 6：移除 HTML 标签

- **来源**：很多 RSS 摘要中嵌入了 HTML 标签（`<p>`、`<em>`、`<a>`、`<img>` 等）。这些对翻译无意义，且有可能干扰模型的 tokenizer。先要求模型删掉它们，只翻译纯文本。
- **局限**：如果原文依赖 HTML 结构传达信息（如列表 `<li>`），移除后可能丢失结构。MVP 阶段这是一个可接受的妥协。

#### 规则 7：仅返回翻译结果

- **必要性**：模型的输出会直接存入 translations 表并展示给用户。如果模型回复包含「以下是翻译：」或附加说明，用户会在界面上看到这些废话。这条规则是界面一致性的基础。
- **副作用**：无法让模型解释翻译依据。如果需要调试翻译质量，用户应临时修改 prompt，加回解释要求。

## 用户自定义 System Prompt 指引

在 Cento 设置页中，用户可以修改 System Prompt。以下是给用户的指引文字：

### 你可以自定义的内容

- **专业领域术语表**：如果你的订阅源集中在特定领域（如医学、法律、计算机），可以在 Prompt 中添加该领域的术语对照表。例如：
  ```
  医学翻译术语表：
  - "adverse event" → "不良事件"
  - "endpoint" → "终点指标"
  - "hazard ratio" → "风险比"
  ```

- **翻译风格**：可以指定更口语化或更书面化的风格。例如：
  ```
  我喜欢简洁的口语化翻译，用短句。
  ```

- **摘要长度**：如果你希望摘要更短或更长，可以调整规则 4。例如：
  ```
  摘要翻译控制在 100 个汉字以内。
  ```

### 不建议修改的内容

- **规则 2（人名不翻译）、规则 7（仅返回翻译结果）**：这两个规则与界面的信息展示强耦合，修改后可能导致显示异常或信息丢失。
- **角色设定**：保持「学术与新闻翻译助手」的角色定义，修改它为其他角色（如「文学翻译」）会导致风格不一致。

### Prompt 调试技巧

如果翻译效果不理想，按以下步骤排查：

1. **先用默认 Prompt 测试**：确认是 Prompt 的问题还是原文本身难以翻译。
2. **单条规则修改**：每次只改一条规则，测试效果后再改下一条。同时改多条无法定位问题。
3. **对比测试**：用同一个 RSS 条目，分别用旧 Prompt 和新 Prompt 翻译，对比结果。
4. **注意 token 消耗**：更长的 System Prompt 会消耗更多 token（System Prompt 内容每次都作为上下文发送）。建议控制在 500 字以内。

## 自动翻译行为

抓取新文章后，后台管线（`services::translation_pipeline`）自动对未翻译的标题和摘要做并发翻译，并发数受信号量控制。标题与摘要共用同一套 System Prompt，区别在于输入文本长度不同。翻译结果写入 `translations` 表缓存，之后打开文章详情时直接展示译文。
