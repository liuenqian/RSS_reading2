# PubMed 检索批次管理设计

日期：2026-07-15
状态：规格审查通过，待用户复核

## 1. 背景与目标

Cento 已支持 PubMed RSS、标题与摘要翻译、阅读状态、期刊指标、阅读笔记和单篇/多篇文献对话。用户还会主动在 PubMed 中构建专题检索，并希望直接在 Cento 中完成检索、增量更新、快速筛选和后续阅读，不再经过“网页导出文件 -> 上传工具 -> 再导入应用”的中间步骤。

本功能的目标是增加“持续 PubMed 检索批次”：用户以自然语言生成并确认 PubMed 检索式，应用通过 NCBI 官方 E-utilities 抓取结果；后续可按同一检索式检查更新，只追加新文献并保留既有人工记录。

## 2. 已确认的产品原则

1. PubMed 检索批次与 RSS 订阅并列管理，但复用同一套文献详情、翻译、笔记和对话能力。
2. 检索批次负责保存检索式、更新和筛选；“保留文献”负责汇总所有批次中值得后续阅读的文章。
3. 同一文献只有一个规范文献记录。它可以同时属于多个检索批次和 RSS 源。
4. 筛选状态按检索批次分别保存；已读、翻译、标签、笔记和对话按规范文献全局共用。
5. 抓取完成后自动翻译全部标题；只有被标记为“保留”的文献才触发摘要翻译。
6. AI 可以辅助筛选，但只产生建议和理由。未经用户确认，不得修改筛选状态。
7. 更新只追加新结果并补充外部元数据，不删除旧记录，也不覆盖用户内容。

## 3. 范围

### 3.1 本次包含

- 自然语言生成 PubMed 检索式。
- 可编辑的检索式预览、命中数量和少量结果预览。
- 直接调用 NCBI ESearch/EFetch 抓取文献，无需用户导出或上传文件。
- 检索批次的创建、重命名、打开和手动“检查更新”。
- PubMed 结果的分批抓取、进度、取消、失败重试和增量合并。
- 按 PMID 优先、规范化 DOI 次之的跨 RSS/检索批次去重。
- 批次内筛选状态：未筛选、保留、待定、排除。
- 标题翻译、保留后摘要翻译。
- IF、JCR 分区、中科院分区、Top 展示与筛选。
- 发表时间与首次加入批次时间的展示和排序。
- 复用现有单篇/多篇文献对话，并增加结构化 AI 筛选建议的确认应用流程。
- “保留文献”汇总入口。

### 3.2 本次不包含

- 自动定时运行检索。第一版只提供手动“检查更新”。
- 自动接受 AI 筛选结果。
- 自动下载付费全文或绕过机构访问控制。
- 把 PubMed 结果导出为 Excel、NBIB 或其他文件格式。
- 新建独立模型配置、独立聊天系统或第三方文献检索服务。
- 从 My NCBI 私有历史、提醒或收藏中读取数据。

## 4. 用户流程

### 4.1 新建检索

1. 用户在左侧“PubMed 检索”区域点击“新建检索”。
2. 用户输入中文或英文研究问题。
3. 应用复用现有自然语言转 PubMed 检索式能力生成候选检索式。
4. 应用展示可编辑检索式、命中总数和少量结果预览。
5. 用户确认名称和检索式后开始抓取。
6. 应用创建检索批次，通过 NCBI History 分批取得记录并实时展示进度。
7. 文献写入本地后进入标题翻译队列；摘要保留原文，不立即翻译。
8. 完成页显示匹配总数、新增数、复用已有记录数和失败数。

如果 AI 生成检索式失败，用户仍可手动输入 PubMed 检索式。预览失败时不创建批次。

### 4.2 标题初筛

批次列表默认展示未筛选文献。每行依次包含：

1. 复选框。
2. 当前排序下的连续编号。
3. 中文标题和英文原标题。
4. 期刊、发表年月、PMID。
5. IF、JCR 分区、中科院分区和 Top。
6. 当前筛选状态。

连续编号是当前批次、筛选条件和排序结果的展示编号，不写入数据库。AI 对话使用同一编号引用本轮所选文献。

用户可以将单篇或多篇文献设为：

- 未筛选：尚未判断。
- 保留：进入后续阅读，并在缺少有效中文摘要时触发摘要翻译。
- 待定：保留在批次中等待进一步判断，不触发摘要翻译。
- 排除：保留记录和筛选状态，不触发摘要翻译。

从“保留”改为其他状态时，已经生成的摘要翻译继续作为缓存保留，不删除、不重复计费。

### 4.3 AI 辅助筛选

1. 用户勾选若干文献并点击“AI 辅助筛选”，或在现有文献对话中选择“所选文献”。
2. 应用将所选文献的编号、PMID、标题、原始摘要、期刊指标及用户输入的纳入/排除标准提交给现有文献对话服务。
3. AI 返回每篇文献的建议状态和简短理由。
4. 结果先显示在对话中，不修改数据库。
5. 用户点击“查看并应用建议”后，在确认界面逐项检查或调整。
6. 用户最终确认后，以单个事务批量更新当前检索批次中的筛选状态。
7. 新设为“保留”的文献进入摘要翻译队列。

普通自由对话继续支持当前文献、所选多篇和自定义组合，并保持现有追加到阅读笔记的能力。AI 可直接使用英文摘要进行判断，不要求先生成中文摘要。

### 4.4 检查更新

1. 用户在检索批次中点击“检查更新”。
2. 应用使用保存的原检索式重新执行完整 ESearch，并取得当前 PMID 集合。
3. 将当前集合与该批次已关联的文献比较。
4. 新 PMID 分批 EFetch、规范化、去重后加入批次，筛选状态为“未筛选”。
5. 已存在 PMID 更新 `last_seen_at`，只补充缺失或发生变化的 PubMed 元数据。
6. 本次未匹配到的旧记录不删除、不移出批次、不改变筛选状态。
7. 只有全部检索和差异合并成功后才更新批次的 `last_success_at`。
8. 完成后显示：当前匹配数、新增数、已有数和失败数。

采用“完整 PMID 集合做差异”而不是只按日期增量查询，避免漏掉延迟收录、索引修订或补充 PMID 的文献。

## 5. 信息架构与界面

### 5.1 左侧导航

在现有阅读入口下增加：

- 保留文献：汇总任一检索批次中状态为“保留”的规范文献，去重显示。
- PubMed 检索：显示已保存批次、当前总数和最近一次更新新增数。
- 新建检索：打开创建流程。

删除一个检索批次只删除批次及关联/筛选状态。仍属于 RSS、其他批次或拥有用户内容的规范文献不得被删除。

### 5.2 批次页

页头展示批次名称、只读的已确认检索式、上次成功更新时间、检查更新按钮和统计：全部、新增、未筛选、保留、待定、排除。检索式只在首次抓取前可编辑；批次创建后保持不可变，保证每次更新的语义一致。需要修改时使用“复制为新检索”，在预览页编辑并创建新批次，原批次及成员不变。

筛选项：

- 筛选状态。
- 最低 IF。
- JCR Q1-Q4/无数据。
- 中科院 1-4 区/无数据。
- Top。
- 首次加入时间范围。
- 发表时间范围。

排序项：

- 发表时间：最新优先/最早优先。
- 加入批次时间：最新优先/最早优先。
- 影响因子：高到低/低到高。
- PubMed 相关性顺序。

默认按发表时间最新优先。列表采用增量渲染或窗口化，避免大量检索结果一次创建全部 DOM 节点。

### 5.3 时间显示

- `publication_date_raw` 保存 PubMed 原始日期文本。
- `publication_date` 保存可确认的部分 ISO 日期：`YYYY`、`YYYY-MM` 或 `YYYY-MM-DD`。
- `publication_date_precision` 保存 `year | month | day | season | freeform | unknown`。
- `publication_sort_key` 保存数值排序键 `YYYYMMDD`。季节只用于排序时映射为 Spring=03、Summer=06、Autumn/Fall=09、Winter=12，界面仍显示 PubMed 原始季节文本，不把近似月份展示给用户。
- 无可解析日期的文献无排序键，并在正序、倒序中都排在有日期文献之后；同一年中缺少月份/日期的记录也排在日期精度更高的记录之后。
- 列表默认显示到月份，例如“2026年7月”；只有年份时显示“2026年”。
- PubMed 只有季节或 MedlineDate 时保留原始日期文本，同时提供可排序的最佳规范化值。
- `first_seen_at` 是文献第一次加入当前检索批次的时间，后续更新不得重置。

### 5.4 期刊指标

复用 `src/journal-metrics.json` 和现有期刊名称规范化逻辑，展示：

- 影响因子 `if`。
- JCR 分区 `q`。
- 中科院分区 `b`，界面显示为 1-4 区。
- Top `top`。

期刊匹配不到时显示“暂无指标”，不得用相似期刊猜测。指标属于当前本地指标库的展示结果，不写回 PubMed 原始元数据。

## 6. 数据模型

### 6.1 规范文献身份

现有 `entries` 继续作为阅读、翻译、标签、笔记和对话使用的规范文献表。新增统一身份解析规则：

1. 有 PMID 时以标准化 PMID 为最高优先级。
2. 无 PMID 时使用规范化 DOI。
3. 两者都没有时，只在同一来源内使用现有 GUID；不跨来源进行标题模糊合并。

为支持“一篇文献属于多个 RSS 源或检索批次”，迁移时重建 `entries` 表：

- `feed_id` 改为可空，外键删除行为从 `ON DELETE CASCADE` 改为 `ON DELETE SET NULL`。
- 删除 `entries` 上的 `UNIQUE(feed_id, guid)` 所有权约束；该约束迁移到 RSS 成员关系表。
- Rust `Entry.feed_id` 改为可选值，所有按 RSS 源读取的查询改为通过成员关系表连接。
- PubMed-only 文献使用 `feed_id = NULL`，无需创建会被调度器误抓取的隐藏 RSS 源。
- 新增标准化展示列 `pmid_normalized`、`doi_normalized`，并通过独立身份表约束可用于自动匹配的活跃身份。

表重建和重复记录合并必须在单个事务中完成。没有重复身份的 entry ID 原样保留；已有重复记录按以下确定性规则选择规范记录并迁移引用：

1. 先按 `pmid_normalized` 分组；无 PMID 时按 `doi_normalized` 分组。
2. 每组优先选择拥有用户内容（阅读笔记、标签、聊天引用或已读状态）最多的 entry；数量相同则选择最小 entry ID。
3. 所有 RSS 成员关系合并到规范 entry。
4. `is_read` 使用逻辑或，`read_at` 保留最早的非空阅读时间；外部元数据按“非空且信息更完整”合并。
5. 标签取大小写不敏感并集。
6. 翻译按字段保留最新的非空缓存；其他翻译属于可再生成缓存，不覆盖保留项。
7. 不冲突的阅读笔记直接迁移；相同 `profile_id` 的两份笔记按生成时间追加到同一内容中，并加入迁移分隔标题，保证文本不丢失。
8. 聊天 session 的 `entry_ids_json` 重写为规范 entry ID 并重新计算 `scope_key`；多个 session 合并到同一 scope 时，消息按原时间和 ID 顺序合并。
9. `reading_events.entry_id` 重写为规范 entry ID，但每条历史事件本身保留。
10. 所有外键和 JSON 引用迁移成功并通过完整性检查后，才删除重复 entry。

新增 `entry_identifiers` 保存规范身份：

- `entry_id`
- `kind`：`pmid | doi`
- `value_normalized`
- `status`：`active | conflicted`
- `source`
- `created_at`

对 `(kind, value_normalized)` 建立仅覆盖 `status = 'active'` 的唯一索引。PMID 分组完成后，每个规范 entry 的 PMID 身份为 active。DOI 只有在没有关联到多个不同 PMID/规范 entry 时才为 active，并可参与自动去重。

同时对 `(entry_id, kind, value_normalized)` 建立唯一约束，`entry_id` 外键使用 `ON DELETE CASCADE`，避免规范 entry 删除后留下无主身份行。

如果不同 PMID 的多个 entry 共享同一 DOI，或一条记录的 PMID 指向规范 entry A、DOI 却指向规范 entry B，则不得自动合并：

1. 相关 PMID 身份继续 active，两个规范文献保持独立。
2. 该 DOI 的所有身份记录标记为 conflicted，因此不违反 active 唯一索引，也不参与自动匹配。
3. `entries.doi_normalized` 可保留用于显示，但不承担唯一约束。
4. 在 `entry_identity_conflicts` 中记录 DOI、相关 PMID/entry ID、来源和时间。
5. 后续带 PMID 的记录仍按 PMID 复用规范 entry；仅带冲突 DOI 的 RSS 记录不自动合并，保留独立来源记录并追加冲突记录。

### 6.2 检索批次

新增 `pubmed_searches`，至少包含：

- `id`
- `name`
- `question`：用户原始研究问题
- `query`：用户最终确认的 PubMed 检索式
- `created_at`
- `last_attempt_at`
- `last_success_at`
- `last_result_count`

新增 `pubmed_search_entries`，至少包含：

- `search_id`
- `entry_id`
- `screening_status`：`unreviewed | keep | maybe | exclude`
- `first_seen_at`
- `last_seen_at`
- `first_seen_run_id`
- `screened_at`
- `is_current_match`
- `pubmed_rank`

唯一约束为 `(search_id, entry_id)`。同一规范文献在不同批次拥有独立筛选状态。

新增 `pubmed_search_runs` 记录每次首次抓取或更新：

- `id`
- `search_id`
- `started_at`
- `completed_at`
- `status`：`running | completed | partial | failed | cancelled`
- `matched_count`
- `added_count`
- `reused_count`
- `failed_count`
- `error_message`

运行记录用于准确识别“本次新增”、展示结果摘要和恢复失败任务，不保存用户密钥。

新增 `pubmed_search_run_items` 保存每次 ESearch 返回的完整 PMID 快照：

- `run_id`
- `pmid`
- `rank`：该次 PubMed 返回顺序中的位置
- `status`：`pending | fetched | reused | failed`
- `entry_id`
- `error_message`

唯一约束为 `(run_id, pmid)`。ESearch 成功后先一次性持久化快照，再开始 EFetch。部分失败或取消后，错误页面可以列出失败 PMID。

run 状态转换规则：

- 新建后为 `running`，`completed_at = NULL`。
- 用户取消后为 `cancelled`；有部分成功但仍有失败项时为 `partial`；未产生可用快照或结果时为 `failed`。这些终止状态均写入 `completed_at`。
- 用户选择恢复同一个 run 时，将状态改回 `running`、清空 `completed_at`，只把 `failed` 项重置为 `pending`，已 `fetched/reused` 项不重复请求。
- 所有 item 成为 `fetched/reused` 后，run 变为 `completed` 并重新写入 `completed_at`。

只有 run 全部成功时，才在单个事务中用 run items 更新 `pubmed_search_entries.is_current_match/pubmed_rank`、`last_success_at` 和当前匹配统计。本次完整集合中不存在的旧成员设置为 `is_current_match = 0`、`pubmed_rank = NULL`，但成员和筛选状态继续保留。部分、失败或取消的 run 不替换上一次成功排名。

### 6.3 RSS 成员关系

新增 `entry_feed_memberships`，并在迁移时从现有 `entries.feed_id/guid` 回填：

- `entry_id`
- `feed_id`
- `guid`
- `first_seen_at`
- `last_seen_at`

唯一约束为 `(feed_id, guid)`。现有 RSS 抓取在解析出 PMID/DOI 后先解析规范文献身份，再创建或更新成员关系。删除 RSS 源或执行保留期清理时，不得删除仍属于 PubMed 检索批次、其他 RSS 源或拥有用户内容的规范文献。

删除 RSS 源时先删除该源的成员关系，再将仍把该源作为兼容 `feed_id` 的 entry 置空或切换到另一个成员 feed。只有完全没有 RSS 成员、PubMed 批次成员及用户内容的 entry 才可作为孤儿清理；删除 feed 本身不得级联删除规范 entry。

## 7. 后端组件边界

### 7.1 PubMed 检索服务

新增独立的 PubMed 检索批次服务，复用现有 `pubmed_service.rs` 的自然语言转检索式能力和 `article_service.rs` 中已经验证的 ESearch/EFetch 解析经验。职责包括：

- 预览检索式和命中数量。
- 使用 Entrez History 分页取得结果。
- 解析 PubMed XML 为现有 Entry 字段。
- 规范化 PMID、DOI、期刊和日期。
- 创建/更新规范文献、批次成员和运行记录。
- 发送可取消的进度事件。
- 遵守 NCBI 无 API Key 时每秒不超过 3 个请求的限制。

第一版不新增 NCBI API Key 设置。请求使用固定节流、有限重试和退避；不能因为单篇解析失败终止整批任务。

首次创建的 ESearch 顺序和每次成功更新的 ESearch 顺序写入 `pubmed_search_run_items.rank`。批次的“PubMed 相关性”排序使用最近一次成功 run 提交到 `pubmed_search_entries.pubmed_rank` 的值；历史上存在但不再匹配当前查询的成员排在当前匹配结果之后。

### 7.2 身份解析与合并

将跨来源身份解析放在共享服务中，供 PubMed 批次导入和 RSS 抓取复用。合并时：

- 用户内容（已读、翻译、标签、笔记、对话）以现有规范文献为准，不覆盖。
- PubMed 元数据只补空值或更新可验证的外部字段。
- 发现 PMID 与 DOI 分别指向两个旧 entry 时不得静默自动合并；记录冲突并报告。

### 7.3 翻译管线

- 首次加入批次的新文献进入现有标题翻译队列。
- 已有有效标题翻译直接复用。
- 状态首次变为“保留”时，如没有有效摘要翻译则进入现有摘要翻译队列。
- 翻译失败不回滚文献导入或筛选状态；界面提供重试。
- 空字符串仍不得视为有效缓存。

### 7.4 AI 筛选

复用现有 paper chat 的模型、费用记录、消息持久化和多文献上下文构建。新增结构化筛选请求/响应层，但聊天文本仍完整保存，便于回看理由。

AI 返回结果必须按 entry ID/PMID 绑定，展示编号只用于交互，不作为持久化身份。无法匹配到所选文献的建议显示为错误，不得应用。批量应用在一个数据库事务中完成。

## 8. 前端状态与交互

- 新建检索、抓取和检查更新均显示明确进度、当前批次和取消按钮。
- 正在运行的同一检索禁止重复启动。
- 切换筛选或排序后重新计算连续编号。
- 复选框位于编号之前。
- 多选状态不随打开右侧详情而丢失；切换批次时清空选择，避免误操作其他批次。
- AI 筛选确认界面显示编号、标题、PMID、建议状态和理由，允许逐项修改。
- “保留文献”是动态汇总视图，不复制记录；文献仍显示所属检索批次。
- 大批量标题/摘要翻译继续使用现有进度和失败重试交互。

## 9. 错误处理与恢复

- 检索式生成失败：保留输入，允许手工编辑检索式。
- ESearch 预览失败：不创建批次，显示可重试错误。
- 首次抓取部分失败：保留已成功记录，将 run 标记为 `partial`；重试只补缺失 PMID。
- 更新失败：不修改 `last_success_at`，不删除或重置任何旧成员。
- 单篇 XML 解析失败：计入失败数并记录 PMID，继续处理其余文献。
- 期刊指标未匹配：显示“暂无指标”，不影响导入。
- 翻译失败：保留原文和筛选状态，可单独重试。
- AI 响应无法解析或缺少身份：只显示原始回答，不提供“应用建议”。
- 用户取消：已提交事务的数据保留，未完成部分可重新运行；run 标记为 `cancelled`。

任何错误信息都不得包含 API Key、请求授权头或本地敏感路径。

## 10. 数据迁移与兼容性

1. 在单个事务中创建新 `entries` 结构、身份/冲突表、成员关系、批次、run 和 run item 表。
2. 回填标准化 PMID/DOI 和 RSS 成员关系，按 6.1 的规则合并已有重复 entry、标记冲突 DOI 并迁移所有引用。
3. 保持所有非重复 entry ID；重复 entry 只在引用迁移、JSON 重写和完整性检查成功后删除。
4. 用新表替换旧 `entries`，重新创建索引和外键；失败时回滚并阻止应用使用半迁移结构。
5. 现有 RSS 列表、阅读统计、保留期清理和删除订阅源逻辑改为识别多来源成员关系。
6. 迁移前后验证 feeds、规范 entries、translations、reading_notes、paper_chat messages、tags 和 reading events 的内容/引用完整性，而不能只比较原始 entries 行数。
7. 对迁移前数据库创建备份；只有迁移成功并通过 `PRAGMA foreign_key_check` 后才继续启动。

## 11. 测试策略

### 11.1 Rust 单元/集成测试

- PubMed XML：完整日期、仅年月、仅年份、季节、MedlineDate、缺失摘要、多作者、多个 identifier。
- 身份解析：PMID 优先、DOI 兜底、标题不得跨来源模糊合并、身份冲突不得静默合并。
- 身份索引：active PMID/DOI 唯一，冲突 DOI 可多条保存且不得用于自动匹配。
- 数据迁移：规范 entry 选择、标签并集、笔记内容合并、聊天 scope 重写、历史事件重映射、事务回滚。
- 首次抓取：新建、复用 RSS entry、同一 PMID 不重复。
- 增量更新：只追加新文献；旧筛选状态、翻译、笔记和聊天不变。
- 多批次：同一 entry 可在两个批次拥有不同筛选状态。
- 多来源：删除一个 RSS 源或批次不删除仍被其他来源引用的 entry。
- 保留触发摘要翻译一次；已有缓存不重复请求。
- AI 筛选建议在确认前不修改状态，确认后事务性更新。
- 失败/取消 run 的状态、计数和恢复行为。
- run item 快照、仅重试失败 PMID、成功后才替换当前匹配标记和相关性排名。

### 11.2 前端测试/检查

- 复选框、编号、标题和状态顺序稳定。
- 筛选/排序后编号和 AI 引用一致。
- 发表时间和加入时间的正序/倒序正确。
- 仅年/年月/完整日期/季节日期的展示和排序稳定；无日期始终排在最后。
- IF/JCR/中科院/Top 筛选正确，缺失值行为明确。
- 检索批次切换清空选择，详情切换保留同批次选择。
- AI 建议确认界面可逐项修改且不能误应用到其他批次。
- 大结果集增量渲染不冻结窗口。

实施完成后至少运行：

- `node --check src/main.js`
- `cargo test --manifest-path src-tauri/Cargo.toml`
- `cargo check --manifest-path src-tauri/Cargo.toml`
- 与 PubMed 测试检索的本地交互验证，包括首次抓取、更新、筛选、保留后摘要翻译和 AI 建议确认。

## 12. 验收标准

1. 用户无需打开 PubMed 网页或上传文件，即可从自然语言创建可编辑检索式并抓取结果。
2. 同一 active PMID 或无冲突的 active DOI 不会因 RSS、多个检索批次或重复更新产生第二个用户可见文献记录；不同 PMID 共享冲突 DOI 时按冲突规则保留独立记录。
3. 更新后旧筛选状态、已读、翻译、标签、笔记和对话均保留，新文献可明确识别。
4. 全部新文献只自动翻译标题；只有“保留”触发摘要翻译。
5. 文献列表显示复选框、连续编号、发表年月、首次加入时间、IF、JCR、中科院分区和 Top，并支持已确认排序/筛选。
6. AI 可对勾选文献提出带理由的筛选建议，未确认时数据库没有状态变化。
7. 同一文献在不同检索批次可有不同筛选状态，“保留文献”去重汇总。
8. PubMed 请求失败、部分解析失败、翻译失败或用户取消时，已有数据不丢失且任务可重试。
