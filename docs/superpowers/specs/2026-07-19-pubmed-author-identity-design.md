# PubMed 作者身份检索与持续订阅设计

日期：2026-07-19
状态：规格审查通过，待用户书面复核

## 1. 背景与目标

Cento 已支持 PubMed 作者检索、AI 检索式生成、预览评估、种子论文、作者指纹、身份分组和判断持久化。但当前实现仍有两个根本问题：第一，PubMed XML 只保存文章中遇到的第一条单位，无法证明该单位属于目标作者；第二，种子论文只用于给现有结果分组，没有基于确认身份执行第二轮扩展检索。因此，当前流程可以减少同名噪声，却不能稳定保证作者文献查全。

本设计将作者检索改为“受控的查全优先”：第一轮使用安全检索寻找可靠种子，用户确认 3–5 篇后建立可审计的作者指纹，再根据确认过的姓名、目标作者专属单位和稳定共同作者执行一次受控扩展检索。两轮结果按 PMID 合并并按作者身份组审核，后续订阅复用同一指纹，只把新增且证据不足的结果交给用户。

## 2. 已确认的产品原则

1. 默认查全优先，但任何扩展都必须由已确认身份证据触发。
2. 纯首字母作者名不得作为独立 PubMed 检索分支。
3. 单位证据必须来自目标作者节点对应的 `AffiliationInfo`，不能使用论文第一条单位代替。
4. 首次使用推荐确认 3–5 篇种子，最多允许 10 篇。
5. 第二轮检索只运行一次；后续新增可靠证据在下一次订阅更新中生效，不进行无限递归扩展。
6. ORCID、目标作者专属单位和稳定共同作者是身份强证据；研究方向只能辅助排序。
7. AI 负责提出别名、解释机构关系和总结证据，不直接覆盖用户判断。
8. 没有足够证据时必须进入“需要确认”，不能强行确认或排除。
9. 用户审核身份组，不逐篇审核全部候选。
10. 已确认判断和拒绝过的别名必须持久化，后续更新不得遗忘。

## 3. 范围

### 3.1 本次包含

- 解析并保存 PubMed 中每位作者及其对应单位、作者顺序和 ORCID。
- 第一轮安全作者检索。
- 种子论文推荐、确认和指纹稳定性提示。
- 根据确认指纹生成第二轮受控扩展检索式。
- 两轮运行记录、结果合并和 PMID 去重。
- 基于目标作者专属单位、共同作者和时间连续性的身份聚类。
- 四档作者身份状态及身份组批量审核。
- 作者指纹、别名可信状态和用户判断持久化。
- 后续检查更新时复用当前指纹和扩展检索式。
- AI 可选的机构别名分析和身份组证据总结。

### 3.2 本次不包含

- 无限制多轮共同作者图谱扩展。
- 接入新的 ORCID、OpenAlex 或其他第三方身份服务。
- 仅凭研究主题自动确认作者。
- 自动接受 AI 建议或覆盖用户判断。
- 对历史数据库中所有 PubMed 文献一次性联网回填作者结构。
- 把作者身份结论跨不同目标作者或不同检索任务自动共享。

## 4. 总体流程

```text
输入作者姓名和已知单位
        ↓
姓名与单位英文别名候选
        ↓
第一轮安全检索
完整姓名 OR（首字母 AND 已知单位）
        ↓
抓取结构化作者及其对应单位
        ↓
用户确认 3–5 篇种子论文
        ↓
建立作者指纹
姓名变体 + 专属单位 + ORCID + 共同作者 + 年份 + 辅助主题
        ↓
第二轮受控扩展检索
完整姓名 OR（首字母 AND confirmed 单位）OR（首字母 AND 稳定共同作者）
        ↓
两轮结果按 PMID 合并去重
        ↓
作者身份聚类与四档判断
        ↓
保存指纹和用户决定，供后续订阅更新复用
```

如果第一轮没有可确认种子，流程停留在种子选择阶段。用户可以调整姓名或单位，也可以通过 PMID 手工加入已知论文。系统不得在没有种子的情况下自动执行第二轮扩展。

## 5. 结构化作者数据

### 5.1 XML 解析

解析每个 PubMed `Author` 节点时保存：

- 作者顺序。
- `LastName`、`ForeName`、`Initials`、`CollectiveName`。
- 规范展示姓名。
- `Identifier Source="ORCID"`，如存在。
- 当前作者节点下全部 `AffiliationInfo/Affiliation`，保持原始顺序。

文章级 `entries.author` 和 `entries.affiliation` 继续保留作为兼容展示字段。`entries.affiliation` 可以展示全部去重单位或第一条单位，但不能再直接参与作者身份判定。

### 5.2 数据表

新增 `pubmed_entry_authors`：

- `id`
- `entry_id`
- `author_order`
- `last_name`
- `fore_name`
- `initials`
- `collective_name`
- `display_name`
- `orcid`
- 唯一约束：`(entry_id, author_order)`

新增 `pubmed_entry_author_affiliations`：

- `id`
- `entry_author_id`
- `affiliation_order`
- `raw_text`
- 唯一约束：`(entry_author_id, affiliation_order)`

两个表均随规范 `entry` 级联删除。重复抓取同一 PMID 时按作者顺序重建该文章的作者结构，重建与文章元数据更新放在同一事务中；解析失败不得删除上一份有效结构。

### 5.3 历史数据兼容

旧记录没有结构化作者数据时标记为“作者单位对应关系未知”。打开作者身份审核或参与新的作者检索更新时，可以随正常 EFetch 结果惰性回填；不能把旧 `entries.affiliation` 自动升级为目标作者专属单位。

### 5.4 目标作者节点解析

单位、ORCID 和共同作者证据必须先定位到论文中的目标作者节点。解析按以下固定优先级执行：

1. 首次建立指纹时使用用户输入的姓名变体和第一轮检索使用的单位别名：先按完整姓名签名匹配，再按姓和首字母匹配。
2. 同级有多个作者时，优先保留单位匹配者；单位仍相同或缺失时不得自动选择。
3. 用户确认种子时，如果仍有多个候选节点，种子卡片必须展开作者列表和每位作者的单位，要求用户明确点选目标作者节点。
4. 用户点选的节点成为该篇种子的唯一目标节点，并可提取其姓名、ORCID、单位和共同作者证据。
5. 后续种子使用已确认姓名签名和 ORCID 优先解析；再次出现并列节点时仍要求用户选择，不能沿用第一作者默认值。
6. 如果指纹已有 confirmed ORCID，唯一 ORCID 精确匹配的作者节点就是目标节点；否则使用 confirmed 完整姓名签名匹配。姓名签名保存规范 token 顺序、反转顺序、姓和给定名首字母；全名匹配优先于首字母匹配。
7. 如果只达到首字母级别，必须再有 confirmed 单位匹配或稳定共同作者支持，并且只有一个节点满足，才可选择。没有匹配、多个节点并列最高，或同一论文中存在无法消解的同名作者时，目标节点状态为 `unresolved`。

`unresolved` 文献仍可进入候选列表，但不得从中提取目标作者专属单位、ORCID或共同作者证据，自动状态只能是“需要确认”。不同非空 ORCID 永远不视为同一作者；ORCID 规范化时去除 URL 前缀、空格和连字符并统一大小写。

## 6. 作者指纹

现有 `pubmed_author_identity_states` 继续按检索任务保存 `state_json`，将 `schema_version` 升级并扩展为：

- 目标作者原始名称及规范形式。
- 用户确认的种子 entry ID 和 PMID。
- 姓名变体：值、来源、可信状态。
- 单位别名：原文、规范名、来源、可信状态、首次和最后出现年份。
- ORCID 集合。
- 稳定共同作者及出现次数。
- 辅助研究主题词。
- 指纹稳定性、版本号和最近计算时间。
- 身份组决定及用户确认时间。
- 稳定身份签名 UUID、签名证据及其用户决定。
- 第一轮和第二轮检索式快照。

别名可信状态为：

- `confirmed`：来自用户确认种子中目标作者对应字段，或用户明确确认。
- `inferred`：AI 或规则推断的英文别名、旧称或上下级机构关系，只可用于提示和候选排序。
- `rejected`：用户明确否定，后续不能再次自动加入。

只有 `confirmed` 姓名、单位和共同作者证据可以进入第二轮检索式。

稳定身份签名首次由用户确认身份组时生成一个不可变 UUID，并保存规范化签名快照。签名序列化为版本号、目标姓名签名、ORCID、排序后的 confirmed 单位规范 ID 和排序后的稳定共同作者签名，以 `|` 连接并去除空白后保存；后续新增证据更新签名快照但不改变 UUID。旧的派生组 key 只在迁移时映射一次，迁移后所有组级决定都引用该 UUID。

证据晋升规则：

- 用户确认种子，或把论文/身份组明确设为“确认作者”时，如果目标作者节点可唯一解析，则该节点的完整姓名、ORCID 和专属单位晋升为 confirmed。
- “高度可能”不会自动晋升别名，只增加待用户复核的候选证据。
- inferred AI 别名只有在用户明确接受，或在已确认论文的目标作者字段中再次出现后，才能晋升为 confirmed。
- 共同作者在两篇及以上 confirmed 论文中出现后成为稳定共同作者；单篇出现只保留为候选，不能触发共同作者扩展。
- 用户确认同名身份组时，记录该组的 ORCID、专属单位和共同作者组合为 rejected 身份签名；不得因为同名组而全局拒绝可能也属于目标作者的姓名缩写。
- confirmed/rejected 证据集合发生变化时，指纹版本加一并重算扩展查询。当前流程不自动触发第三轮抓取，新版本从下一次检查更新开始使用。

## 7. 两轮检索策略

### 7.1 第一轮安全检索

第一轮目标是获得可靠种子，不追求一次查全。检索式由后端策略编译，AI 只返回结构化候选值。

```text
完整姓名变体[Full Author Name]
OR
（首字母姓名[Author] AND 已知单位英文别名[Affiliation]）
```

规则：

- 完整姓名变体可以独立检索。
- 首字母姓名必须与至少一个英文单位别名绑定。
- 中文单位只保留为用户输入和显示值，PubMed 检索使用可审阅的英文别名。
- 日期范围继续由应用精确追加。
- 生成后再次执行语法检查和纯首字母安全检查。

### 7.2 种子推荐

推荐候选按以下顺序排序：

1. ORCID 或完整姓名可识别。
2. 目标作者对应单位与已知单位相符。
3. 作者、单位、年份等元数据完整。
4. 能提供稳定共同作者和跨年份证据。

默认推荐 3–5 篇，最多允许用户选择 10 篇。确认 1–2 篇可以建立“有限指纹”，但所有扩展结果默认需要确认；至少 3 篇且覆盖两类独立证据后才可显示“稳定”并启用完整扩展。

### 7.3 第二轮受控扩展

确认种子后生成。扩展能力按种子证据分级：

- 确认 1–2 篇时执行有限扩展：完整姓名，以及“首字母 AND confirmed 目标作者单位”；不使用共同作者分支，所有新增结果默认需要确认。
- 确认至少 3 篇且覆盖两类独立证据后执行完整扩展：再加入稳定共同作者分支和 confirmed 历史单位。

```text
完整姓名及 confirmed 拼写变体
OR
（首字母姓名 AND confirmed 当前或历史单位）
OR
（首字母姓名 AND 稳定共同作者）
```

稳定共同作者必须在两篇及以上确认种子中与目标作者共同出现。单篇共同作者只保留为候选，不能触发共同作者扩展。

有限扩展不包含最后一条共同作者分支；只有完整扩展才使用三条分支。

第二轮不得使用 `inferred` 或 `rejected` 别名，不得加入主题词，不得生成纯首字母分支。第一轮和第二轮 PMID 取并集，保留各自来源轮次和 PubMed 排名。

### 7.4 运行记录

一次作者检索更新是一个逻辑 run，可以包含一个或多个查询分支。复用 `pubmed_search_runs` 和 `pubmed_search_run_items`，新增 `pubmed_search_run_queries`：

- `id`
- `run_id`
- `query_kind`：`base | expansion`
- `query`
- `profile_version`
- `status`：`pending | completed | failed`
- `result_count`
- `error_message`

新增 `pubmed_search_run_item_sources` 保存同一 PMID 在各分支中的来源和排名：

- `run_id`
- `pmid`
- `run_query_id`
- `rank`
- 唯一约束：`(run_query_id, pmid)`

`pubmed_search_run_items` 仍保存整个逻辑 run 的 PMID 并集，唯一约束保持 `(run_id, pmid)`。每次运行同时保存：

- 运行类型：`initial | expansion | refresh`。
- 使用的作者指纹版本。

首次创建的 `initial` run 只有 `base` 分支。确认种子后的 `expansion` run 和后续 `refresh` run 都执行当时版本的 `base` 与 `expansion` 两个分支，在同一 run 内合并去重。一个分支失败时 run 标记为 partial，不更新当前匹配集合；恢复时只重试失败分支和未完成 PMID。只有所有查询分支和全部 PMID 都完成后，才在一个事务中提交 PMID 并集并更新 `is_current_match`。

同一 PMID 命中两个分支时只保存一篇文献，但保留两条来源记录。展示排序优先使用 base 分支排名；只命中 expansion 的结果排在 base 结果之后，并使用 expansion 排名。这样不会发生第二轮完成后把第一轮独有文献错误隐藏的问题。

## 8. 身份证据与聚类

### 8.1 硬性规则

- ORCID 与目标一致：最高强度身份证据，但最终“确认作者”状态仍遵守用户确认规则。
- ORCID 明确不同：`同名作者`。
- 用户确认过的论文或身份组：保持用户状态。
- 姓名不匹配任何 confirmed 变体：不能自动确认。
- 单位、摘要或 ORCID 缺失不算冲突。

### 8.2 证据组合

没有 ORCID 时：

- 目标作者专属单位匹配：强证据。
- 稳定共同作者网络匹配：强证据。
- 姓名全称、缩写和罗马化匹配：基础证据。
- 单位变化与发表年份连续：辅助证据。
- 研究方向连续：辅助证据，只用于排序和解释。

自动状态：

- 姓名匹配且至少两类独立强证据一致：`高度可能`。
- 姓名匹配且只有一类强证据：`需要确认`。
- 只有姓名或主题相近：`需要确认`。
- ORCID 冲突，或至少两类明确负面证据：`同名作者`。
- `确认作者` 除 ORCID 一致外，必须由用户确认。

明确负面证据仅包括：

- 目标作者节点 ORCID 与 confirmed ORCID 不同。
- 目标作者节点单位只匹配 rejected 身份签名，且不匹配任何 confirmed 单位。
- 共同作者集合与某个 rejected 身份签名有至少两个稳定共同作者重合，同时与目标指纹没有稳定共同作者重合。
- 完整给定名与所有 confirmed 姓名签名不兼容；仅缺少全名或只有首字母不算不兼容。

缺失值、单位变化、共同作者没有重合、研究方向变化和年份超出既有范围都不是负面证据。

### 8.3 身份聚类

聚类前先规范化特征：姓名使用与目标节点解析相同的签名；单位先匹配 confirmed/rejected 别名，再移除通用机构词进行 token 比较；共同作者使用规范姓名签名并排除目标作者。

单位匹配满足下列任一条件：

- 两个单位映射到同一个 confirmed 规范单位。
- 规范文本完全相同。
- 双方至少各有两个有效特征 token，较短 token 集是较长集合的子集，或 Jaccard 相似度不低于 0.65。只共享 hospital、university、department、medical、China 等通用词不得匹配。

共同作者强匹配满足下列任一条件：

- 至少两个规范共同作者签名重合。
- 至少一个完整姓名共同作者重合，且 Jaccard 相似度不低于 0.30。

候选文献先按相同非空 ORCID 强制归入同一组；不同非空 ORCID 永远禁止合并。其余候选按 PMID 稳定排序后构建无向证据图。姓名签名兼容且满足“单位匹配”或“共同作者强匹配”的两篇文献建立边；confirmed/rejected 身份冲突时禁止建立边。身份组是该图的连通分量，因此输入顺序不影响结果。无 ORCID 节点无法唯一归入某一子组时单独进入“需要确认”组。

组内展示分数主要由目标作者专属单位、共同作者网络和时间连续性组成。默认相对权重为：

- 专属单位：60%。
- 共同作者网络：30%。
- 时间连续性：10%。

时间连续性只提供正向加分：发表年份位于指纹范围前后五年内记为连续，超出范围不算冲突。主题不进入聚类边和主要分数，只在组内候选分数相同的情况下作为稳定排序参考。

### 8.4 用户决定与重新聚类

派生的 affiliation/coauthor 组键不作为持久身份。组级操作在事务中写入当前成员的逐篇状态，并创建稳定身份签名 UUID；签名保存当时的 ORCID、confirmed 单位和稳定共同作者证据。

重新聚类后：

- 旧成员保留逐篇用户状态，不被自动算法改写。
- 新身份组只在恰好匹配一个稳定身份签名且没有冲突时继承该签名决定。
- 一个旧组拆分后，已有成员保留状态；新成员分别按各子组证据判断。
- 多个旧组发生合并且决定不一致时，已有成员保持原状态，整个新组和新增成员显示“需要确认”。
- 新组同时匹配多个稳定签名时不得继承，进入“需要确认”。

## 9. AI 职责边界

勾选 `AI 评估` 后，AI 在用户确认种子并完成结构化作者解析后运行。职责包括：

- 提议姓名和机构英文别名。
- 解释机构简称、旧称、不同翻译和上下级关系。
- 汇总每个身份组的一致证据、冲突和缺失信息。
- 对 `需要确认` 身份组给出下一步应核对的具体字段。

AI 不得：

- 把 inferred 别名直接升级为 confirmed。
- 仅凭主题确认作者。
- 因单位缺失判定同名作者。
- 修改用户已确认状态。
- 生成或采用纯首字母宽检索。

AI 不可用时，结构化解析、两轮查询策略、规则评分和人工审核仍可工作。

## 10. 用户界面

### 10.1 首次检索

1. 用户输入作者姓名、已知单位和可选日期范围。
2. 应用展示第一轮检索式和命中数。
3. 抓取后展示推荐种子候选，用户确认 3–5 篇；有歧义时先在作者列表中点选目标作者节点。
4. 应用显示指纹稳定性并按种子数量自动运行有限或完整第二轮。
5. 完成后显示两轮新增、复用、合并后的总数和身份组统计。

### 10.2 身份组审核

身份组列表显示：

```text
身份组 | 文献数 | 目标作者对应单位 | 常见共同作者 | 年份 | 研究方向 | 判断
```

展开后显示组内论文、目标作者对应单位和逐项证据。每组可批量设置为确认作者、高度可能、需要确认或同名作者。同名作者默认隐藏但不删除。

### 10.3 后续订阅更新

- 使用一个 refresh run 同时执行当前 base 和 expansion 查询，并在全部分支成功后提交 PMID 并集。
- 复用当前 confirmed 指纹和对应版本检索式；本轮中新确认的证据只影响下一次 refresh。
- 新结果自动进入已有身份组或新的待审核组。
- 确认作者的新论文自动纳入。
- 高度可能默认纳入，但保留批量复核标记。
- 需要确认进入独立审核列表。
- 用户决定和 rejected 别名持续保留。

## 11. 错误处理与恢复

- AI 失败：保留输入和候选别名，允许手工填写英文单位并继续规则流程。
- 第一轮无种子：停止扩展，允许调整查询或通过 PMID 加入已知论文。
- PubMed 请求失败：复用现有 run 快照和分页恢复，只重试未完成 PMID。
- 作者结构解析失败：保留文章记录和上一份有效作者结构，将该文献降级为需要确认。
- 第二轮查询生成失败：第一轮结果和指纹继续保留，允许重新生成或手工审核。
- 结果量较大：后台分页抓取并增量更新身份组，不一次性渲染全部行。
- 数据冲突：降级为需要确认，不覆盖用户历史决定。
- 旧记录缺少作者结构：显示证据缺口，不使用文章级第一单位替代。

## 12. 迁移与兼容性

1. 新增作者和作者单位关联表，不重建现有 `entries`。
2. 作者身份状态使用新的 `schema_version`，读取旧 JSON 时迁移已有 seed IDs、状态 IDs、单位别名和组决定。
3. 旧检索继续按原查询运行；用户进入作者身份流程并确认种子后，才生成扩展查询。
4. 现有文章级作者和单位列继续显示，表格与导出格式保持兼容。
5. 只有新抓取或惰性回填的文章拥有作者级单位证据；旧数据不得被静默视为 confirmed。
6. 数据库迁移前继续使用现有备份机制，迁移失败必须回滚。

## 13. 测试与验收

### 13.1 XML 与数据层

- 目标作者不是第一作者时，正确读取目标作者自己的单位。
- 首次种子存在多个同名作者时，必须由用户点选目标作者节点。
- 同一论文中多个作者、多个单位和多单位作者正确关联。
- ORCID、集体作者、缺失全名和缺失单位正确解析。
- 重抓同一 PMID 更新作者结构且不会留下重复行。
- 新解析失败时不删除旧的有效结构。

### 13.2 查询策略

- 中文单位生成可审阅的英文别名候选。
- 所有首字母分支都带单位或稳定共同作者约束。
- 没有 confirmed 种子或 confirmed 目标作者单位时不生成第二轮；1–2 篇种子只生成有限扩展。
- inferred 和 rejected 别名不进入第二轮查询。
- 第一轮与第二轮结果按 PMID 正确合并去重。
- refresh 的 base/expansion 分支必须在一个逻辑 run 中原子提交；单分支失败不能隐藏上一轮结果。
- 日期范围在两轮查询中保持一致。

### 13.3 身份判断

- 当前单位英文全称、简称和 confirmed 历史名称可归入同一身份。
- 作者换单位但共同作者和时间连续时不会直接判为同名。
- 同名作者拥有冲突 ORCID 时明确分离。
- 相同非空 ORCID 的论文强制归入同一身份组。
- 1–2 篇种子只生成有限扩展，至少 3 篇且具备两类证据后才生成共同作者扩展。
- 单位缺失、摘要缺失或主题变化不会单独触发排除。
- AI 错误建议不能修改 confirmed/rejected 状态。
- 用户确认过的身份组在更新后保持判断。
- 旧派生组 key 迁移为稳定身份 UUID，重新聚类后决定继承符合第 8.4 节。

### 13.4 集成与性能

- 从首次检索、种子确认、第二轮抓取到身份组审核完整通过。
- 取消、失败和恢复不会重复抓取已完成 PMID。
- 每批最多渲染 100 行，抓取和聚类阶段持续报告批次进度。
- 1,000 篇合成候选的结构化评分与聚类测试在 CI 中应在 2 秒内完成，算法不得对全部候选执行无索引的全量两两 AI 比较。
- 用户取消应在当前网络请求或当前本地批次完成后生效，不启动下一批。
- `cargo check`、相关 Rust 测试、`node --check src/main.js` 和作者身份前端测试通过。

验收完成标准：纯首字母噪声不再出现；自动判断均展示可审计证据；证据不足时不会强行分类；用户主要审核身份组而非逐篇审核；后续订阅能够复用已确认指纹并只突出新增低置信结果。

## 14. 实施顺序

1. 增加结构化作者与单位数据表及 XML 解析测试。
2. 将作者身份算法切换到目标作者专属单位，移除文章第一单位的判定权重。
3. 扩展作者指纹 schema 和可信别名状态。
4. 实现第二轮安全查询编译和运行快照。
5. 串联种子确认、自动扩展、合并去重和身份组审核。
6. 接入可选 AI 解释并完善失败恢复。
7. 完成迁移、集成和性能验证。
